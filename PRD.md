## 🎯 วัตถุประสงค์ (Objectives)

### **วัตถุประสงค์หลัก:**
1. รวบรวมข้อมูลลูกค้าจากหลายช่องทาง (LINE, โทร, Walk-in, Facebook)
2. ติดตามสถานะงานแบบ Real-time ทั่วทั้งทีม
3. จัดการ Workflow การทำงานที่ซับซ้อน (นัดคิว, ใบเสนอราคา, เช็คอะไหล่)
4. เก็บข้อมูลและรูปภาพไว้ในระบบ Centralized
5. ไม่มีค่าใช้จ่ายในการดำเนินการ (100% Free)

### **ปัญหาที่แก้ไข:**
- ❌ ข้อมูลลูกค้ากระจัดกระจาย (ไม่มีที่รวม)
- ❌ ไม่รู้ว่างานอยู่ที่สถานะไหน
- ❌ การสื่อสารระหว่างฝ่ายผิดพลาด
- ❌ ต้องจดข้อมูลซ้ำๆ
- ❌ ไม่มีประวัติการทำงาน

### **รายละเอียดแต่ละเส้นทาง:**

#### **📍 เส้นทาง A: นัดคิว**
```
1. แอดมินรับเรื่อง → บันทึกข้อมูลลูกค้า
2. เลือกสถานะ → "ขอคิว"
3. แอดมินทำงานจริง: นัดคิวช่าง ลงปฏิทิน
4. กลับมาเปลี่ยนสถานะ → "ลงปฏิทินแล้ว"
5. แจ้งลูกค้า → เสร็จสิ้น
```

#### **📍 เส้นทาง B: ขอใบเสนอราคา**
```
1. แอดมินรับเรื่อง → บันทึกข้อมูลลูกค้า
2. เลือกสถานะ → "ขอใบเสนอราคา"
3. ฝ่ายทำใบเสนอราคาเห็นงาน → ทำใบเสนอราคาจริง
4. ฝ่ายทำใบเสนอราคากลับมาเปลี่ยนสถานะ → "รอส่ง"
5. แอดมินทำงานจริง: ส่งใบเสนอราคาให้ลูกค้า
6. แอดมินกลับมาเปลี่ยนสถานะ → "ส่งลูกค้าแล้ว"
7. เสร็จสิ้น
```

#### **📍 เส้นทาง C: เช็คอะไหล่**
```
1. แอดมินรับเรื่อง → บันทึกข้อมูลลูกค้า
2. เลือกสถานะ → "เช็คอะไหล่เสนอราคา"
3. ฝ่ายจัดซื้อเห็นงาน → เช็คราคาอะไหล่จริง
4. ฝ่ายจัดซื้อเปลี่ยนสถานะ → "ขอใบเสนอราคา"
5. ฝ่ายทำใบเสนอราคาเห็นงาน → ทำใบเสนอราคาจริง
6. ฝ่ายทำใบเสนอราคาเปลี่ยนสถานะ → "รอส่ง"
7. แอดมินทำงานจริง: ส่งใบเสนอราคาให้ลูกค้า
8. แอดมินเปลี่ยนสถานะ → "ส่งลูกค้าแล้ว"
9. เสร็จสิ้น
```

## 📊 สถานะงาน (Status)

### **8 สถานะหลัก:**

| รหัสสถานะ | ชื่อสถานะ | ไอคอน | คำอธิบาย |
|-----------|-----------|--------|-----------|
| `new` | รับเรื่องใหม่ | 📥 | งานใหม่ที่เพิ่งรับเข้ามา |
| `queue` | ขอคิว | 📋 | รอนัดคิวช่าง |
| `scheduled` | ลงปฏิทินแล้ว | 📅 | นัดคิวเสร็จแล้ว |
| `quotation` | ขอใบเสนอราคา | 💰 | กำลังทำใบเสนอราคา |
| `check_parts` | เช็คอะไหล่เสนอราคา | 🔧 | กำลังเช็คราคาอะไหล่ |
| `waiting_send` | รอส่งลูกค้า | 📤 | ทำเสร็จแล้ว รอส่ง |
| `sent` | ส่งลูกค้าแล้ว | ✅ | เสร็จสิ้นทั้งหมด |
| `cancelled` | ยกเลิก | ❌ | ยกเลิกงาน |

**หมายเหตุ:** ทุกฝ่ายเห็นงานทั้งหมด (ไม่กรองตามฝ่าย)

### **การเปลี่ยนสถานะ (Status Transitions):**

```javascript
new         → queue, quotation, check_parts, cancelled
queue       → scheduled, cancelled
scheduled   → sent, cancelled
quotation   → waiting_send, check_parts, cancelled
check_parts → quotation, cancelled
waiting_send→ sent, cancelled
sent        → (เสร็จสิ้น)
cancelled   → (เสร็จสิ้น)
```

## 🎨 ความต้องการฟีเจอร์ (Features)

### **Phase 1: ฟีเจอร์หลัก (MVP - Minimum Viable Product)**

#### **1. ระบบการล็อกอิน (Authentication)**
- ✅ Login ด้วยการเลือกผู้ใช้ (8 คน)
- ✅ แบ่งสิทธิ์ตามฝ่าย (admin, quotation, procurement)

#### **2. ระบบรับเรื่อง (Request Management)**
- ✅ สร้างงานใหม่ (ฝ่ายแอดมิน เท่านั้น)
- ✅ กรอกข้อมูลลูกค้า:
  - ชื่อลูกค้า *
  - เบอร์โทร *
  - ที่อยู่
  - ช่องทางติดต่อ (LINE, โทร, Walk-in, Facebook, อื่นๆ)
- ✅ กรอกรายละเอียดงาน:

#### **4. ระบบค้นหาและกรอง (Search & Filter)**

#### **5. Dashboard**
- ✅ แสดงสถิติ:
  - งานทั้งหมด
  - งานที่รอดำเนินการ
  - งานที่เสร็จสิ้น

#### **6. ระบบจัดเก็บข้อมูล (Data Storage)**
- ✅ เก็บข้อมูลใน Google Sheets
- ✅ เก็บรูปภาพ/วิดีโอใน Google Drive

## ⚠️ ข้อจำกัดและความเสี่ยง (Risks & Limitations)

### **ข้อจำกัด:**
1. **Performance** - Google Apps Script อาจช้าถ้าข้อมูลมาก
2. **Customization** - จำกัดการปรับแต่ง UI
3. **Offline** - ต้องออนไลน์เสมอ

### **ความเสี่ยง:**
1. **Google เปลี่ยน Policy** - อาจกระทบต่อระบบ
2. **ข้อมูลเยอะมาก** - อาจช้าลง
3. **User Error** - อาจกรอกข้อมูลผิด

### **วิธีจัดการ:**
1. **Performance** - ใช้ caching และ limit data
2. **Customization** - ยอมรับข้อจำกัด มุ่ง functionality ก่อน
3. **Offline** - ใช้ localStorage เป็น fallback
4. **Google Policy** - ติดตามข่าวสาร Google
5. **ข้อมูลเยอะ** - Archive ข้อมูลเก่า
6. **User Error** - ทำคู่มือชัดเจน
